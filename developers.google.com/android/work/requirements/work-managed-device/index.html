Work-managed device

Overview

The work-managed device solution set is designed for corporate-owned devices that are used exclusively for work and no personal use. This allows admins to manage the entire device and enforce an extended range of policy controls unavailable to work profiles, but restricts the device to only corporate use.

Mandatory features

1. Enterprise provisioning

1.2. DPC identifier device provisioning

(Android 6.0+) Allows users to enter a DPC's unique "afw#" identifier during the initial setup of a device, which automatically installs the DPC. Once installed, the DPC sets itself as device owner.

  • a) End users can provision a work-managed device using a DPC identifier, according to the defined implementation guidelines.
    • i) The EMM DPC must be publicly availble on Google Play, and the end user can install the DPC from the device setup wizard by entering the DPC-specific identifier.
    • ii) Once installed, the DPC must guide the user through the process of provisioning a work managed device.

2. Device security

2.1. Device security challenge

(Android 5.0+) Allows admins to protect work data by mandating a device passcode of a certain type and complexity.

  • a) Enterprise admin can set device security challenge policies (PIN/pattern/password).

2.5. Wipe and lock

(Android 5.0+) Enables admins to remotely lock or wipe work data from the EMM console, if for instance a device is lost or stolen.

  • a) Enterprise admin can lock and wipe work data from the EMM console.

2.6. Compliance enforcement

(Android 5.0+) Ensures that access to corporate data is automatically restricted if the device is not in compliance with basic security policies such as password complexity.

  • a) Access to work data is automatically restricted if a device is not in compliance with security policies.
    • i) Enforced security policies must include (but is not limited to) password policy.

2.7. Default security policies

(Android 5.0+) Ensure that corporate data is secure without any special configuration or customization by admins in their EMM console.

  • a) Unknown sources must be blocked by default.
    • i) Unknown sources may be enabled for debugging purposes, but may not be enabled for enterprise app distribution.
    • ii) EMMs are encouraged (but not required) not to surface policy controls to change the default state of these security features.
  • b) Debugging features must be blocked by default.

3. Google Play app management

3.1. Managed Google Play accounts enterprise enrollment

(Android version N/A) Enables admins to create a corporate entity (called an enterprise) used to distribute apps to devices through managed Google Play. The enrollment flow is integrated into the EMM console to enable admins to quickly configure their EMM for access to Google Play's app management features.

  • a) Enterprise admin can create and enroll a managed Google Play Accounts enterprise.
  • b) The EMM console must silently provision and configure a unique ESA for each enterprise.
  • c) Enterprise admin can un-enroll a managed Google Play Accounts enterprise.

3.2 Managed Google Play account provisioning

(Android 5.0+) Silently creates and provisions enterprise user accounts for accessing the managed Google Play store. These managed Google Play accounts are created and maintained by EMMs to identify managed users, enabling unique, per-user app distribution rules.

  • a) The EMM's DPC can silently provision and activate a managed Google Play account according to the defined implementation guidelines.
    • i) A managed Google Play account of type userAccount must be added to the device.
    • ii) Managed Google Play accounts of type userAccount must have a 1-to-1 mapping with actual users managed in the EMM console.

3.5. Silent app distribution

(Android version N/A) Allows admins to install, uninstall, and update work apps on users' devices without any user interaction.

  • a) Enterprise admin can silently install, update, and uninstall apps using the Play EMM API.

3.6. Managed configuration management

(Android 5.0+) Enables admins to silently configure apps according to configuration parameters unique to each app. For instance, different apps surface parameters such as VPN configurations, email configurations, and app-level security policies.

  • a) Enterprise admin can programmatically retrieve and set managed configurations for any app using managed configurations through Play, according to the defined implementation guidelines.
  • b) After retrieving a managed configuration schema from Play, the EMM must display these managed configuration settings to the enterprise admin in a way that allows them to easily view and set any configuration type (as defined by the Android framework) for any given Play app.
  • c) The EMM must provide wildcards (such as $username$ or %emailAddress%) so that apps such as Gmail can have a single configuration applied correctly to a number of users.

3.7. App catalog management

(Android version N/A) Allows admins to view all apps approved for distribution in their enterprise, including apps purchased in the managed Google Play console, Play private apps only visible to the admin, and apps programmatically approved through the EMM, in a single view within the EMM console.

  • a) Enterprise admin can import the My Company Apps list of apps approved in the managed Google Play console (play.google.com/work), including private apps and paid apps, into the EMM console for distribution to devices.

3.9. Basic store layout management

(Android version N/A) Enables users to install and update work apps on devices through the managed Google Play store on devices. The basic layout lists all apps approved for use within the enterprise in a single list, which EMMs filter on a per-user basis according to policy.

  • a) Enterprise admin can manage users’ available product sets to allow users to view and install applications from the managed Google Play store under the section ‘Store home’.
    • i) The basic store layout is sufficient to meet the layout component of this criteria. EMMs that wish to offer store layout customization should review the advanced store layout configuration feature.
    • ii) In both cases, however, EMMs must manage users’ available product sets to ensure apps are made visible to users.

3.15. Play EMM API usage requirements

(Android version N/A) Ensure that EMMs properly implement the API at scale, and avoid traffic patterns that could negatively impact customers' ability to manage apps in production environments.

  • a) EMMs should adhere to the Play EMM API usage limits. Failure to correct behavior that exceeds these guidelines may result in suspended API access, at Google’s discretion.
  • b) EMMs should distribute traffic from different customers throughout the day, rather than consolidating all customers' traffic at specific or similar times. Behavior that fits this traffic pattern, such as scheduled batch operations for all enrolled customers, may result in suspend API access, at Google's discretion.
  • c) Consistent, incomplete or deliberately incorrect requests that make no attempt to retrieve or manage actual customer data may result in suspended API access, at Google's discretion.

4. Device management

4.1. Runtime permission policy management

(Android 6.0+) Allows admins to set a default runtime permission response (including allow, deny, or allowing users to choose) for all permissions requests made by work apps. The permission policy is silently set for work apps when they request permissions.

  • a) Enterprise admin can set a runtime permission policy for each of the policy constants: prompt, grant, and deny.

4.2. Runtime permission grant state management

(Android 6.0+) Allows admins granular control over individual permissions for individual work apps. After setting a default runtime permission policy, admins can adjust the settings for specific permissions on specific apps.

  • a) Enterprise admin can set the grant state of any permission of any work app (built on API 23 or above) to each of grant states: default, grant, and deny.

N/A

Optional features

1. Enterprise provisioning

1.3. NFC device provisioning

(Android 5.0+) Allows admins to "bump" new or factory-reset devices with an EMM-supplied provisioning app. This bump starts the device provisioning process, and passes a number of parameters (such as server details) to the DPC to streamline the setup process.

  • a) Enterprise admin can provision a work-managed device for single use using an EMM-specific NFC provisioning app published in Play, according to the defined implementation guidelines.
    • i) Using provisioning extras, the provisioning app must pass all registration details (for instance server IDs, enrollment IDs) aside from unique user or device credentials (for instance username/password, activation token), so that users don't need to enter those details when activating a device. EMMs must not include sensitive information, such as passwords or certificates, in the registration details.
    • ii) The DPC must launch immediately once set as device owner, and lock itself to the screen until the device is fully enrolled, to prevent users from leaving the provisioning process.

1.4. QR code device provisioning

(Android 7.0+) Allows admins to scan an EMM-supplied QR code using new or factory reset devices. The code begins the device provisioning process and specifies a number of parameters such as server details that are passed to the DPC to streamline the setup process.

  • a) Enterprise admin can provision a work-managed device for single use using the QR code provisioning method from a QR code generated by the EMM console, according to the defined implementation guidelines.
    • i) Using provisioning extras, the QR code must pass all registration details (for instance server IDs, enrollment IDs) aside from unique user or device credentials (for instance username/password, activation token), so that users need not enter those details when activating a device. EMMs must not include sensitive information, such as passwords or certificates, into the registration details.
    • ii) The DPC must launch immediately once set as device owner, and lock itself to the screen until the device is fully enrolled, to prevent users from leaving the provisioning process.

1.5. Zero-touch enrollment

(Android 8.0+ (Pixel 7.1+)) Allows admins to preconfigure devices purchased from authorized resellers for management with their EMM. When the device is first turned on, the device will be forced into management according to the configuration specified by the admin.

  • a) Enterprise admin can provision a work-managed device using the zero-touch enrollment method, outlined in the help center article Zero-touch enrollment for IT admins.

1.6. Advanced zero-touch enrollment

(Android 8.0+ (Pixel 7.1+)) Allows admins to automate much of the device enrollment process by deploying DPC registration details through zero-touch enrollment. The DPC supports limiting enrollment to specific accounts or domains, according to the configuration options offered by the EMM.

  • a) Enterprise admin can provision a work-managed device using the zero-touch enrollment method, outlined in the help center article Zero-touch enrollment for IT admins.
  • b) The DPC must launch immediately once set as device owner, and lock itself to the screen until the device is fully enrolled, to prevent users from leaving the provisioning process. The DPC need not be locked to the screen in the event that the device is fully provisioned silently via the zero-touch enrollment registration details. For example, in a COSU deployment.
  • c) Using the registration details, the DPC must ensure that only authorized users are permitted to proceed with activation once the DPC is invoked. At a minimum, activation must be locked down to all users of a given customer.
  • d) Using the registration details, the DPC must make it possible for admins to pre-populate all registration details (for instance server IDs, enrollment IDs) aside from unique user or device credentials (for instance username/password, activation token), so that users need not enter those details when activating a device.
    • i) EMMs must not include sensitive information, such as passwords or certificates, in the zero-touch enrollment's configuration.
  • e) The EMM must define and report to Google a registration schema that accomplishes the requirements from above. The EMM can report the schema at registration time or during product review

1.8. Google Account device provisioning

(Android 6.0+) For customers that use G Suite, this feature guides users through the installation of their EMM's DPC after they enter their corporate G Suite credentials during initial device setup. Once installed, the DPC completes the setup of a work-managed device.

  • a) End users can provision a work-managed device using the Google Account provisioning method, according to the defined implementation guidelines.
  • b) Unless the EMM can unequivocally identify the device as a corporate asset owned by the enterprise customer, the device cannot be provisioned without prompting the user during the provisioning process to take a non-default action such as checking a checkbox or selecting a non-default menu choice. It is recommended that the EMM is able to identify the device as a corporate asset so that the user does not need to be prompted.

2. Device security

2.3. Advanced passcode management

(Android 5.0+) Enables admins to manage advanced password lifecycle features like password expirations, as well as the full set of alphanumeric configuration options available in Android.

  • a) Enterprise admin can configure a different password quality type for each lock screen available on the device. Enterprise admins must be able to choose from any of the following requirement types:
    • i) Unspecified: Sets no password requirement.
    • ii) Something: Requires a password but doesn't set a type restriction.
    • iii) Weak biometric: Allows low-security biometric unlock methods, such as face recognition.
    • iv) Numeric: Requires a password that includes numeric characters.
    • v) Numeric complex: Requires a password that includes numeric characters and has no repeating (for example 4444) or ordered sequences (for example 1234).
    • vi) Alphabetic: Requires a password that includes alphabetic or other symbol characters.
    • vii) Alphanumeric: Requires a password that includes both numeric and alphabetic (or other symbol) characters.
    • viii) Complex: Requires a password that includes a numeric, alphabetic, and special character.
  • b) Enterprise admin can set separate password length and complexity requirements, for each lock screen available on the device. Enterprise admins must be able to set any of the following requirements::
    • i) Minimum length required for a valid password.
    • ii) Minimum number of numeric characters required for a valid password.
    • iii) Minimum number of non-letter characters required for a valid password.
    • iv) Minimum number of letters required for a valid password.
    • v) Minimum number of lower case letters required for a valid password.
    • vi) Minimum number of upper case letters required for a valid password.
    • vii) Minimum number of symbols required for a valid password.
  • c) Enterprise admin can deploy different password lifecycle configurations for each lock screen available on the device. Enterprise admins must be able to choose from any of the following options:
    • i) Password expiration timeout: Coupled with the compliance enforcement feature, this forces the user to periodically update their password according to the admin-specified timeout. Admins must be able to disable this feature.
    • ii) Password history length: Specifies the length of time before a user can re-use any given password. Admins must be able to disable this feature.
    • iii) Maximum failed passwords for wipe: Specifies the number of times the user can enter an incorrect password before corporate data is wiped from the device. Admins must be able to disable this feature.

2.4. Smart Lock management

(Android 6.0+) Allows admins to control what trust agents in Android's Smart Lock feature are permitted to unlock devices. Admins can disable specific unlock methods like trusted bluetooth devices, face recognition, and voice recognition, or optionally disable the feature entirely.

  • a) Enterprise admin can disable all Smart Lock trust agents, independently for each lock screen available on the device.
  • b) Enterprise admin can selectively enable and configure Smart Lock trust agents, independently for each lock screen available on the device, for all of the following trust agents: bluetooth, NFC, places, face, on-body, and voice.
    • i) Enterprise admin can modify all of the available trust agent configuration parameters.

2.9. SafetyNet support

(Android version N/A) Gives admins confidence in the devices managed by their EMM by leveraging Google's SafetyNet Attestation API to validate the integrity of the device and ensure it is a valid Android device.

  • a) The EMM's DPC implements the SafetyNet attestation API during provisioning, before adding corporate data to a device, by confirming that the basicIntegrity and ctsProfileMatch values return true.
    • i) The EMM's console must verify the compatibility check according to the defined implementation guidelines.
  • b) EMM DPC sends a compatibility check every time the device checks in with the EMM server, configurable by the enterprise admin, and verifies the APK information in the compatibility check response as well as response itself before updating corporate policy on the device
  • c) Enterprise admin can configure different policy responses based on the values returned by SafetyNet, including blocking provisioning or wiping corporate data.
    • i) By default, access to corporate data should be blocked if the device or requesting APK fails any validation, according to the compliance enforcement feature requirements.

2.10. Verify Apps enforcement

(Android 5.0+) Gives admins confidence that users' devices are protected against harmful apps by ensuring Android's Verify Apps feature is enabled. Verify Apps scans apps installed on Android devices for malware before and after they're installed, to ensure corporate data cannot be compromised by malicious apps.

  • a) Enterprise admin can enforce that verify apps is enabled on a managed device.
    • i) This feature must be enabled by default for all devices.

2.11. Direct Boot support

(Android 7.0+) Ensures that the EMM's DPC is always active and able to enforce policy, even if parts of the system have not yet been decrypted following a reboot. Since Android 7.0, apps cannot run on devices that have just been powered on until the device is first unlocked. Direct Boot support ensures the DPC can run and access data even if the device has not been unlocked.

  • a) The EMM's DPC leverages device encrypted storage to perform critical management functions before the DPC's credential encrypted storage has been decrypted. Management functions available during Direct Boot must include (but are not limited to):
    • i) Enterprise wipe, including the ability to wipe factory reset protection data as appropriate.
  • b) The EMM's DPC must not expose corporate data within device encrypted storage.

2.12. Hardware security management

(Android 5.1+) Enables admins to lock down hardware elements of the device to ensure data-loss prevention, such as preventing data transfer via usb, NFC beam, and physical media like SD cards.

  • a) Enterprise admin can block users from mounting physical external media on their device.
  • b) Enterprise admin can block users from sharing data from their device using NFC beam.
  • c) Enterprise admin can block users from transferring files over USB from their device.
  • d) Enterprise admin can toggle whether USB storage is enabled or disabled on users' devices.

2.13. Enterprise security logging

(Android 7.0+) Enables admins to gather usage data from devices, which can be parsed and programmatically evaluated for malicious or risky behavior. Activities logged include Android Debug Bridge (adb) activity, app launches, and screen unlocks.

  • a) Enterprise admin can enable security logging for target devices, and the EMM's DPC will retrieve both security logs and pre-reboot security logs automatically.
  • b) Enterprise admin can review enterprise security logs, for a given device and configurable time window, in the EMM console.
  • c) Enterprise security logs can be exported from the EMM console.

3. Google Play app management

3.8. Programmatic app approval

(Android version N/A) Embeds Google Play's app discovery and approval capabilities inside the EMM console using the Google Play iframe. Admins can search for apps, and approve them without leaving the EMM console, as well as accept new permissions for apps they previously approved.

  • a) Enterprise admin can search for and approve apps within the EMM console using the managed Google Play iframe.
  • b) When an app requires additional permission to be approved on update, enterprise admin can approve these new permissions using the permissions iframe.

3.10. Advanced store layout configuration

(Android version N/A) Enables admins to customize the store layout users see in the managed Google Play store on their devices. Through the EMM console, admins can configure separate pages of app and app clusters, connected by customized links. Advanced store layouts improve discoverability of corporate apps and provide a better user experience than the basic layout.

  • a) Enterprise admin can configure and deploy a custom store layout, including the following customization parameters:
    • i) Enterprise admin can create up to 100 store layout pages. Pages can have an arbitrary number of localized page names.
    • ii) Enterprise admin can create up to 30 clusters for each page. Clusters can have an arbitrary number of localized cluster names.
    • iii) Enterprise admin can assign up to 100 apps to each cluster.
    • iv) Enterprise admin can add up to 10 quick links to each page.
    • v) Enterprise admin can specify the sort order of apps within a cluster and clusters within a page.

3.11. App license management

(Android version N/A) Enables admins to view the number of licenses purchased, consumed, and available for apps purchased at play.google.com/work within the EMM console. Admins can also assign licenses to users silently without requiring users to install the app, to ensure the right users always have access to their apps.

  • a) Enterprise admin can view the number of licenses purchased, licensed consumed, and licenses available for all paid apps approved for the enterprise, as well as the users who have consumed those licenses.
  • b) Enterprise admin can silently assign licenses to users without forcing that app to be installed on any of the users' devices.
  • c) Enterprise admin can unassign an app license from a user.

3.12. Google-hosted private app management

(Android version N/A) Simplifies the Google-hosted private app publishing and update workflows available to all admins through managed Google Play, by enabling admins to update Google-hosted private apps through the EMM console instead of through the Google Play console.

  • a) Enterprise admin can upload new versions of apps that are already published privately to the enterprise using the Google Play Developer Publishing API.

3.13. Self-hosted private app management

(Android version N/A) Guides admins through the process of configuring and publishing apps that can only be accessed by users in that enterprise, but unlike Google-hosted private apps the APKs are not hosted by Google Play. Instead, the EMM helps the enterprise admin host the APK themselves, but still protects the app by ensuring it can only be installed when authorized by managed Google Play.

  • a) The EMM console must assist the enterprise admin in hosting the app APK, by offering both of the following options:
    • i) Hosting the APK on the EMM's server. The server can be on-premise or cloud-based.
    • ii) Hosting the APK outside of the EMM's server, at the discretion of the customer. The enterprise customer must specify in the EMM console where the APK is hosted.
  • b) The EMM console generates an appropriate APK definition file using the provided APK and guides the enterprise admin through the publishing process.
  • c) Enterprise admin can update the self-hosted private app, and the EMM console will silently publish the updated APK definition file using the Google Play Developer Publishing API.
  • d) The EMM server only serves download requests for the self-hosted APK that contain a valid JWT within the request's cookie, as verified by the private app's public key.
    • i) To facilitate this, the EMM server must guide the enterprise admin to download the self-hosted app's license public key from the Play Developer Console, and upload this to the EMM console.

3.14. EMM pull notifications

(Android version N/A) Instead of periodically querying Play to identify past events such as an app update that contains new permissions or managed configurations, EMM pull notifications proactively notify EMMs of such events in real time, enabling EMMs to take automated actions and provide custom admin notifications based on these events.

  • a) EMM must use Play's EMM notifications to pull notification sets.
  • b) EMM must automatically notify enterprise admin (for instance by an automated email) of the following notification events:
    • i) newPermissionEvent: Requires an enterprise admin to approve new app permissions before the app can be silently installed or updated on users' devices.
    • ii) appRestrictionsSchemaChangeEvent: May require enterprise admin to update an app's managed configuration to maintain intended functionality.
    • iii) appUpdateEvent: May be of interest to enterprise admins who wish to validate that core workflow functionality is unaffected by app update.
    • iv) productAvailabilityChangeEvent: May impact users' ability to install the app or take app updates.
    • v) installFailureEvent: Play is unable to silently install an app on a users' device, suggesting there may be something about the device configuration that is preventing the install. EMMs should not immediately retry a silent install after receiving this notification, as Play's own retry logic will have already failed.
  • c) EMM automatically takes appropriate actions based on the following notification events:
    • i) newDeviceEvent: During device provisioning, EMMs must wait for newDeviceEvent before making subsequent Play EMM API calls for the new device, including silent app installs and setting managed configurations.
    • ii) productApprovalEvent: upon receiving a productApprovalEvent notification, EMM must automatically update the list of approved apps imported into the EMM console for distribution to devices if their is an active admin session, or if the list of approved apps is not automatically reloaded at the start of each admin session.

4. Device management

4.3. WiFi configuration management

(Android 6.0+) Enables admins to silently provision corporate WiFi configurations on users' devices, including credentials for basic authentication.

  • a) Enterprise admin can silently provision on devices a WiFi configuration that can include:
    • i) SSID
    • ii) password

4.4. WiFi security management

(Android 6.0+) Enables admins to provision enterprise WiFi configurations that include advanced security features such as authentication certificates and CA certificates.

  • a) Enterprise admin can silently provision on devices a WiFi configuration that can include:
    • i) identity
    • ii) certificate for client authorization
    • iii) CA certificate or certificates

4.5. Advanced WiFi management

(Android 6.0+) Enables admins to lock down WiFi configurations on managed devices, to prevent users from creating new configurations or modifying corporate configurations.

  • a) Enterprise admin can lock down corporate WiFi configurations in either of the following configurations:
    • Users cannot modify any WiFi configurations provisioned by the EMM, but may add and modify their own user-configurable networks (for instance personal networks).
    • Users cannot add or modify any WiFi network on the device, limiting WiFi connectivity to just those networks provisioned by the EMM.

4.6. Account management

(Android 5.0+) Enables admins to ensure that only authorized corporate accounts can interact with corporate data, for services such as SaaS storage and productivity apps, or email. Without this feature, users can add personal accounts to those corporate apps that also support consumer accounts, enabling them to share corporate data with those personal accounts.

  • a) Enterprise admin can prevent users from adding or modifying accounts.
    • i) When enforcing this policy on a device, EMMs must set this restriction before provisioning is complete, to ensure users cannot circumvent this policy by adding accounts before the policy is enacted.

4.7. G Suite account management

(Android 5.0+) Enables admins to ensure that only authorized G Suite accounts can interact with corporate data. Without this feature, users can add personal Google Accounts to corporate Google apps (for example Google Docs or Google Drive), enabling them to share corporate data with those personal accounts.

  • a) Enterprise admin can specify G Suite account(s) to be activated by the user during provisioning, after the account management lockdown is in place.
  • b) The EMM's DPC must prompt the user to activate their G Suite account, and ensure that only the specific user account can be activated by specifying the account to be added.
    • i) On devices below Android 7.0, the DPC must temporarily disable the account management restriction before prompting the user.

4.8. Certificate management

(Android 5.0+) Enables admins to deploy identity certificates and certificate authorities to managed devices, to enable access to corporate resources.

  • a) Enterprise admin can install user identity certs generated by their PKI on a per-user basis.
    • i) The EMM console must integrate with at least one PKI and distribute certificates generated from that infrastructure.
  • b) Enterprise admin can install certificate authorities in the managed keystore.

4.9. Advanced certificate management

(Android 7.0+) Streamlines app setup on devices by allowing admins to silently select which certificates should be used by which managed apps. Furthermore, this feature grants admins the ability to remove CAs and identity certs from active devices, and prevent users from modifying credentials stored in the managed keystore.

  • a) Enterprise admin can, for any application distributed to devices, specify a certificate to which that app will be silently granted access during runtime.
    • i) Certificate selection must be sufficiently generic so to enable a single configuration to apply to all users, each of which may have a user-specific identity certificate.
  • b) Enterprise admin can silently remove certificates from the managed keystore.
  • c) Enterprise admin can silently uninstall a CA certificate, or all non-system CA certificates.
  • d) Enterprise admin can prevent users from configuring credentials in the managed keystore.

4.10. Delegated certificate management

(Android 6.0+) Allows admins to distribute a third-party certificate management app to devices and grant that app privileged access to install certificates into the managed keystore.

  • a) Enterprise admin can specify a certificate management package to be set as the delegated certificate management app by the DPC.
    • i) Console may optionally suggest known certificate management packages, but must allow enterprise admin to choose from the list of all apps available for install, for applicable users.

4.11. Advanced VPN management

(Android 7.0+) Allows admins to specify an always-on VPN to ensure that data from specified corporate apps will always go through a configured VPN. Note that this feature requires deploying a VPN client that supports both always-on and per-app VPN features.

  • a) Enterprise admin can specify an arbitrary VPN package to be set as an always-on VPN by the DPC.
    • i) Console may optionally suggest known VPN packages that support always-on VPN, but cannot restrict the VPNs available for always-on configuration to any arbitrary list.
  • b) Enterprise admin can specify what apps deployed to their users should go through VPN by configuring the VPN though managed configurations

4.13. Advanced IME management

(Android 5.0+) Enables admins to control what input methods (IMEs) users can configure for their device. Advanced IME management extends the basic feature by enabling admins to manage access to system IMEs as well, which are typically provided by the OEM or carrier of the device.

  • a) Enterprise admin can configure an IME whitelist of arbitrary length (excluding an empty list, which blocks all IMEs including system IMEs), which may contain any arbitrary IME packages.
    • i) Console may optionally suggest known or recommended IMEs for whitelisting, but must allow enterprise admin to choose from the list of all apps available for install, for applicable users.
  • b) EMM must prevent admins from configuring an empty whitelist, as this will block all IMEs including system IMEs from being configured on the device.
  • c) EMM must ensure that if an IME whitelist does not contain system IMEs, that the third-party IMEs are silently installed before the whitelist is applied on the device.

4.14. Accessibility services management

(Android 5.0+) Enables admins to control what accessibility services can be enabled on users' devices. While accessibility services are powerful tools for users with disabilities or that are temporarily unable to fully interact with their device, they may interact with corporate data in ways that are non-compliant with corporate policy. This feature allows admins to disable any non-system accessibility service.

  • a) Enterprise admin can configure an accessibility service whitelist of arbitrary length (including an empty list, which blocks all non-system accessibility services), which may contain any arbitrary accessibility service package.
    • i) Console may optionally suggest known or recommended accessibility services for whitelisting, but must allow enterprise admin to choose from the list of all apps available for install, for applicable users.

4.16. Advanced location sharing management

(Android 5.0+) Enables admins to enforce a given location sharing setting on a managed device. This feature can ensure, for example, that corporate apps always have access to high accuracy location data, or that users don't consume extra battery by restricting location settings to battery saving mode.

  • a) Enterprise admin can set the device location services to each of the following modes:
    • i) High accuracy.
    • i) Sensors only, for instance GPS, but not including network-provided location.
    • i) Battery saving, which limits the update frequency.
    • iv) Off.

4.17. Factory reset protection management

(Android 5.1+) Enables admins to protect corporate-owned devices from theft by ensuring only authorized users can factory reset devices. Admins can also disable factory reset protection entirely, in the event it introduces operational complexities when devices are returned to IT.

  • a) Enterprise admin can prevent users from factory resetting their device from Settings.
  • b) Enterprise admin can specify corporate unlock account(s) authorized to provision devices after a factory reset.
    • i) This account can be tied to an individual, or used by the entire enterprise to unlock devices.
  • c) Enterprise admin can disable factory reset protection for specified devices.
  • d) Enterprise admin can initiate a remote device wipe that optionally wipes reset protection data, thereby removing factory reset protection on the reset device.

4.18. Advanced app control

(Android 5.0+) Enables admins to prevent the user from uninstalling or otherwise modifying managed apps through Settings, for instance force closing, the app or clearing an app's data cache.

  • a) Enterprise admin can block uninstall of any arbitrary managed apps, or all managed apps.
  • b) Enterprise admin can prevent users from modifying application data from Settings. disallow app control

4.19. Screen capture management

(Android 5.0+) Enables admins to block users from taking screenshots when using managed apps. This includes blocking screensharing apps and similar apps (such as Google Assistant) that leverage the system screenshot capabilities.

  • a) Enterprise admin can prevent users from capturing screenshots.

4.20. Disable cameras

(Android 5.0+) Enables admins to disable use of device cameras by managed apps.

  • a) Enterprise admin can disable use of device cameras by managed apps.

4.22. Advanced network statistics collection

(Android 6.0+) Enables admins to query network usage statistics for the entire managed device. The statistics collected reflect the usage data shared with users in the Data usage section of settings, and is applicable to usage of all apps on the device.

  • a) Enterprise admin can query for the network statistics summary for the entire device, for a given device and configurable time window, and view this detail in the EMM console.
  • b) Enterprise admin can query a summary of app network usage statistics, for a given device and configurable time window, and view this detail organized by UID in the EMM console.
  • c) Enterprise admin can query the network usage historical data, for a given device and configurable time window, and view this detail organized by UID in the EMM console.

4.23. Reboot device

(Android 7.0+) Enables admins to remotely reboot managed devices.

  • a) Enterprise admin can remotely reboot a managed device.

4.24. System radio management

(Android 7.0+) Enables admins granular control over system network radios and associated usage policies. For instance, admins can prevent users from modifying mobile network and data roaming settings, sending SMS messages, and connecting to bluetooth devices.

  • a) Enterprise admin can disable cell broadcasts sent by service providers (for example AMBER alerts).
  • b) Enterprise admin can prevent users from modifying mobile network settings in Settings.
  • c) Enterprise admin can prevent users from resetting all network settings in Settings.
  • d) Enterprise admin can configure if the device permits cellular data while roaming.
  • e) Enterprise admin can configure whether the device can make outgoing phone calls, excluding emergency calls.
  • f) Enterprise admin can configure whether the device can send and receive SMS messages.
  • g) Enterprise admin can prevent users from using their device as a portable hotspot by tethering.
  • h) Enterprise admin can set the WiFI timeout to the default, only while plugged in, or never.
  • i) Enterprise admin can prevent users from configuring or modifying existing bluetooth connections.

4.25. System audio management

(Android 5.0+) Enables admins to silently control device audio features, including muting the device, preventing users from adjusting volume settings, and preventing users from unmuting the device microphone.

  • a) Enterprise admin can silently mute managed devices.
  • b) Enterprise admin can prevent users from modifying device volume settings.
  • c) Enterprise admin can prevent users from unmuting the device microphone.

4.26. System clock management

(Android 5.0+) Enables admins to control device clock and timezone settings, and prevent users from modifying automatic device settings.

  • a) Enterprise admin can enforce system auto time, preventing the user from setting the date and time of the device.
  • b) Enterprise admin can silently toggle both auto time and auto time zone.

5. Device usability

5.1. Managed provisioning customization

(Android 7.0+) Enables admins to modify the default managed provisioning flow UX to include enterprise-specific features. Optionally, admins can display EMM-provided branding during provisioning.

  • a) Enterprise admin can customize the provisioning process by specifying the following enterprise-specific details: enterprise color, enterprise logo, enterprise terms of service and other disclaimers.
  • b) Enterprise admin can deploy a non-configurable, EMM-specific customization that includes the following details: EMM color, EMM logo, EMM terms of service and other disclaimers.
    • i) EMM must include all provisioning terms of service and other disclaimers for their provisioning flow in the system provisioning disclaimers bundle, even if the EMM-specific customization is not used.
    • ii) EMM may set their non-configurable, EMM-specific customization as the default for all deployments, but must allow admins to configure their own customization.

5.3. Advanced enterprise customization

(Android 7.0+) Enables admins to customize aspects of the managed device with corporate branding, for instance by setting the primary user icon to the corporate logo, or setting the device wallpaper.

  • a) Enterprise admin can set the display name for the managed device.
  • b) Enterprise admin can set the user icon of the managed device.
  • c) Enterprise admin can prevent the user from modifying the device user icon.
  • d) Enterprise admin can set the device wallpaper.
  • e) Enterprise admin can prevent the user from modifying the device wallpaper.

5.4. Lock screen messages

(Android 7.0+) Enable admins to set a custom message that's always displayed on the device lock screen, and does not require device unlock to be viewed.

  • a) Enterprise admin can set a custom lock screen message.

5.5. Policy transparency management

(Android 7.0+) Enables admins to customize the help text provided to users when they attempt to modify managed settings on their device, or deploy an EMM-supplied generic support message. Both short and long support messages can be customized, and are displayed in instances such as attempting to uninstall a managed app for which an admin has already blocked uninstallation.

  • a) Enterprise admin can customize both the short and long support messages.
  • b) Enterprise admin can deploy non-configurable, EMM-specific short and long support messages.
    • i) EMM may set their non-configurable, EMM-specific support messages as the default for all deployments, but must allow admins to configure their own messages.

5.8. System update policy

(Android 6.0+) Enables admins to configure when managed devices take over-the-air software updates. Admins can force devices to take updates immediately when available, postpone updates up to 30 days, or ensure that updates occur only at specific times (for instance to update kiosks only after a store has closed).

  • a) Enterprise admin can set a system update policy of each type: automatic, windowed, and postpone.

5.10. Persistent preferred activity management

(Android 5.0+) Allows admins to set an app as the default intent handler for intents that match a certain intent filter. For example, this would allow admins to choose which browser app automatically opens all web links, or which launcher app is used when the user hits the home button.

  • a) Enterprise admin can set any package as the default intent handler for any arbitrary intent filter.
    • i) The EMM console may optionally suggest known or recommended intents for configuration, but cannot restrict intents to any arbitrary list.
    • ii) The EMM console must allow the enterprise admin to choose from the list of all apps that are available to install for applicable users.

5.12. Advanced keyguard feature management

(Android 5.0+) Enables admins greater control over the device keyguard (lock screen), for instance blocking all notifications in addition to just unredacted notifications, and disabling the secure camera.

  • a) Enterprise admin can disable the following device keyguard features:
    • i) Secure camera.
    • ii) All notifications.
    • iii) Unredacted notifications.
    • iv) Trust agents.
    • v) Fingerprint unlock.
    • vi) All keyguard features.

5.13. Remote debugging

(Android 7.0+) Allows admins and help centers to retrieve debugging resources from users' devices, without requiring the user to take additional steps. Bug reports are powerful tools for diagnosing issues, but may include sensitive data and thus require user consent.

  • a) Enterprise admin can remotely request a bug report, view the bug report within the EMM console, and download the bug report from the EMM console.

5.14. MAC address retrieval

(Android 7.0+) Enables EMMs to silently fetch a device's MAC address, to be used to identify devices in other parts of the enterprise infrastructure (for example when identifying devices for network access control).

  • a) Device MAC address is silently retrieved and associated with the device in the EMM console.
